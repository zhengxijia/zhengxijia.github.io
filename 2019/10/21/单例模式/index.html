简介

单例模式（Singleton Pattern）应该是设计模式中最简单的设计模式之一，但由于java语言的特性（指令重排序），使得单例模式还是有点东西可探讨研究。

应用场景

  1. Windows中的任务管理器，系统同时只能打开一个任务管理器
  2. SpringMVC默认Controller、Service、Dao都是单例模式scope="singleton"，可配置为多例模式scope="prototype"（不推荐），单例模式中的bean通过spring的IOC容器进行管理，使用时可使用依赖注入方式DI获取对象（底层使用ThreadLocal变量保证多线程安全性），不需要每次请求都创建对象。
  3. 数据库连接池、线程池等池子一般都为单例模式，创建一个池子消耗资源太多，比如数据库连接池的创建需要产生和数据库连接的IO
  4. …

简单实现

  • 饿汉模式：类加载的时候便进行创建（加载类时较慢，运行时获取对象较快，线程安全）
  • 懒汉模式：要使用的时候才进行创建（加载类时较快，运行时获取对象较慢，线程不安全）

线程不安全

线程不安全步骤解析：

假设有两个线程同时调用了getInstance()方法.

  1. 当线程1执行到if(instance == null)时, 判断为true, 进入if内, 此时线程1时间片结束, CPU切换到线程2.
  2. 此时线程2执行到if(instance == null), 因为线程1还未来得及执行instance = new Singleton(), 所以if判断为true, 进入方法体, 此时线程2时间片结束, CPU切换到线程1.
  3. 此时线程1执行instance = new Singleton(), instance的hashcode为101, 然后return, 执行结束, 切换回线程2.
  4. 此时线程2在if内, 执行instance = new Singleton(), instance的hashcode为102, 然后return.

上面步骤在莫名其妙的过程中就创建了两个Singleton的实例. 这时候就变成了多例的单例模式？？？

那么进化一下, 加上synchronized同步吧

线程安全（双重锁）

线程安全步骤解析：

假设有两个线程同时调用了getInstance()方法.

  1. 当线程1执行到if(instance == null)时, 判断为true, 进入if内, 此时线程1时间片结束, CPU切换到线程2.
  2. 此时线程2执行到if(instance == null), 因为线程1还未来得及执行instance = new Singleton(), 所以if判断为true, 进入方法体, 此时线程2时间片结束, CPU切换到线程1.
  3. 线程1拿到Singleton.class锁, 进入synchronized, 此时就算切换回线程2, 线程2也会因为拿不到Singleton.class锁对象, 阻塞在同步代码块外面.
  4. 线程1继续执行if, 执行instance = new Singleton(), instance的hashcode为101, 然后return, 执行结束, 切换回线程2.
  5. 此时线程2拿到Singleton.class锁, 进入synchronized, if判断为false, 直接return.

  • 为什么不把第一个if去掉, 直接留个synchronized + if呢？

    因为synchronized为重量级锁（JDK1.5版本出现轻量级锁Lock）, 耗时长且消耗性能，双重锁在多线程情况下能保持高性能.

volatile 优化（JDK1.5后最佳实践）

  • volatile关键字用来解决指令重排序在多线程下的问题, 它有两个功能.
    1. 保证内存可见性
    2. 防止指令重排序

关于内存可见性（其实这已经被synchronized解决了）

Java内存模型规定, 变量存储在主存中, 每个线程拥有该变量的一个拷贝副本在自己的工作内存中, 线程修改变量是修改自己工作内存中的变量, 而修改完毕后, 会将自己工作内存中的修改后的值回写到主存中.

执行步骤再次分析：

  1. 当线程1执行到if(instance == null)时, 判断为true, 进入if内, 拿到Singleton.class锁, 进入synchronized, 继续执行if, 执行instance = new Singleton(), instance的hashcode为101, 此时instance没有回写到主存, 切换回线程2.
  2. 线程2执行到第一个if, 因为线程1的值没有回写到主存, 所以还是会进入if内, 但是被synchronized阻塞了.
  3. 线程1将变量回写到主存, 并return.
  4. 此时线程2拿到Singleton.class锁, 进入synchronized, if判断为false, 直接return.
    所以就算没有使用volatile保证内存可见性, 也不会导致出错.

关于指令重排序

instance = new Singleton() 这条命令并不是原子性的, 它包含3个操作.

  1. 在堆内存申请Singleton实例的内存空间.
  2. 初始化Singleton实例.
  3. 将instance变量指向内存中的Singleton实例的内存空间(执行完这步 instance 就为非 null 了)
    由于JVM的指令重排序, 步骤123可能会变成132, 多线程下会导致出错.

假设有两个线程同时调用了getInstance()方法. 指令顺序为132.

  1. 当线程1执行到if(instance == null)时, 判断为true, 进入if内, 拿到Singleton.class锁, 进入synchronized, 继续执行if, 执行instance = new Singleton().
  2. 先在堆内存申请Singleton实例的内存空间, 由于指令重排序, 将instance变量指向没有初始化的, 但是已经申请了的内存空间. 此时线程1时间片结束, CPU切换到线程2.
  3. 线程2执行到第一个if, 因为instance已经指向没有初始化的内存空间, 所以直接return. 这时调用这个单例, 因为还未初始化, 所以会导致错误出现.

而volatile可以防止指令重排序, 让指令严格按照123的顺序执行.
