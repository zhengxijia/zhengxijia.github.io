前言

缓存的使用，极大得提高了请求的效率，特别是在高并发大流量的情境下，可以有效降低数据库访问压力，但使用缓存也会存在一系列问题，比如数据一致性，但严格意义上来讲，数据一致性问题并不能真正可以解决，类比CAP原则，高一致性与高可用性不可兼得。

抛开不可完美解决的一致性问题，其他比较经典的问题就是缓存穿透、缓存雪崩和缓存击穿。

缓存一般流程

缓存穿透

概念

查询一个一定不存在的key，由于缓存不命中所以需要访问数据库，在请求流量较大时或者有心人士故意利用不存在的key频繁访问，将导致数据库CPU和内存负载过高，甚至可能宕机。

解决方案

  1. 简单粗暴式（推荐）：如果一个查询返回的数据为空（不管是数据不存在，还是系统故障），仍然把这个空的结果进行缓存，即将key-value写为key-null，但要设置较短过期时间，如30秒，最好不要超过5分钟
  2. 布隆过滤器：将所有可能存在的数据哈希到一个足够大的bitmap中，一个一定不存在的数据会被这个bitmap拦截

缓存雪崩

概念

数据未加载至缓存中，或者在某个时间段，缓存集体失效（过期时间相同），所有请求直接访问数据库，导致数据库CPU和内存负载过高，甚至可能宕机。

解决方案

  1. 根据key的类型不同，设置不同的过期时间，让缓存的失效时间点尽量均匀，热点key缓存时间稍长，非热点key缓存时间稍短
  2. 通过缓存命令reload机制，在高流量并发访问前手动触发加载更新缓存
  3. 数据库前设置分级缓存（双缓存策略），A1为原始缓存，A2为拷贝缓存，A1失效，访问A2，A1缓存失效时间短，A2缓存失效时间长

缓存击穿

概念

热点key在缓存过期的一瞬间，此时大量的请求由于热点key缓存过期失效，直接击穿缓存，直接访问数据库，数据库请求量骤增，CPU和内存负载骤增

解决方案

  1. 设置热点key缓存时间永不过期（推荐）
  2. 定时任务或者异步线程主动检查更新缓存过期时间（原理同1）
  3. 数据库前设置分级缓存（双缓存策略）
  4. 加mutex key互斥锁或使用消息队列（不建议，用户体验不好）
